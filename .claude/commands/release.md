Ship a cc-dashboard release end-to-end.

Optional argument passed verbatim to `scripts/release.sh`: `$ARGUMENTS`
- **No arg** → script auto-bumps the patch digit (e.g. `0.1.2` → `0.1.3`).
- **`X.Y.Z`** → use that exact version. **Minor / major bumps must be passed explicitly; never auto-bump minor or major.** If the user typed `/release` without a version and the change set looks like it might warrant a minor bump (new feature, breaking-ish behavior), ask before proceeding.

## Preflight

1. `git status --short`. If dirty, commit pending work first — do not let `release.sh` see a dirty tree (its preflight will fail). Group commits by theme (feature / fix / docs / build / refactor). Don't bundle unrelated edits.
2. **Every commit MUST use identity override** (project rule in CLAUDE.md — the global `user.email` on this machine is a work address that would leak into public history):
   ```
   git -c user.email=pandaleecn@gmail.com -c user.name="Panda Lee" commit -m "..."
   ```
   Never `git config --global …` — only `-c` inline per invocation.
3. **No `Co-Authored-By: Claude …` trailers.** Ever. Not as a default, not as convention. Undoing them retroactively requires history rewrite + force-push.
4. Quit any running dev cc-dashboard:
   ```
   osascript -e 'tell application "cc-dashboard" to quit'
   ```
   Otherwise the smoke test in `release.sh` Step 4 collides on port 7788.
5. Confirm tests are green before shipping: `swift test --parallel`. If red, stop — don't release a broken build.

## Release

Run `./scripts/release.sh $ARGUMENTS` with a generous Bash timeout (≥ 600000 ms). Apple notarize typically takes 2–5 min and the script blocks on it.

The script (in order):
1. Bumps `CFBundleShortVersionString` + `CFBundleVersion` in `Info.plist`.
2. Promotes `CHANGELOG.md`: `## [Unreleased]` stays, a new `## [X.Y.Z] — YYYY-MM-DD` section is inserted right below. The compare-link + release-tag link at the bottom are updated.
3. Commits `Info.plist` + `CHANGELOG.md` as `Bump version to X.Y.Z` (done by script itself — do not pre-commit these).
4. Clean build with `CC_NOTARIZE=1 ./make-bundle.sh` — Developer ID sign + Apple notarytool submit + staple + DMG.
5. Smoke tests the signed app and DMG.
6. Regenerates `docs/appcast.xml` + `homebrew-cask/Casks/cc-dashboard.rb`; commits as `vX.Y.Z: appcast + cask`.
7. `git push origin main` + `git push origin vX.Y.Z`.
8. `gh release create` with the DMG attached.
9. Syncs `homebrew-cc-dashboard` tap repo.
10. Verifies `https://heypandax.github.io/cc-dashboard/appcast.xml` returns 200.

## After

Report to the user:
- Version shipped (from `Info.plist` or the tag).
- GitHub Release URL (printed in Step 8 output).
- Upgrade path: `brew upgrade --cask cc-dashboard` or in-app "Check for Updates…" (Sparkle).

## Hard rules

- **Force-push to main requires explicit per-incident authorization.** `release.sh` never force-pushes; if you ever find yourself about to, stop and ask.
- Do not edit `docs/appcast.xml` or `homebrew-cask/Casks/cc-dashboard.rb` by hand — they are regenerated by `scripts/update_appcast.sh`.
- Never skip notarize (`--skip-notarize`) for a real release. That flag exists for script debugging and produces an unshippable DMG.
- If any step fails (notarize rejection, push conflict, tap sync failure), stop and report. Recovery may need manual `git tag -d vX.Y.Z && git reset --hard HEAD~2` to unwind the two release commits.
